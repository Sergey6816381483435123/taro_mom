## Context

Сайт — статический GitHub Pages-проект без сборщика. Сейчас существуют основная русская страница `/`, английская `/en/` и по четыре тематические страницы на русском и английском. Главные страницы уже используют self-canonical, взаимные `hreflang`, JSON-LD, доступный HTML и общую визуальную систему. Бот подтверждённо поддерживает русский, английский, немецкий, испанский, французский, хинди, португальский и турецкий.

Изменение охватывает сначала основные лендинги. Полная локализация тематических страниц — отдельная следующая итерация после получения данных о спросе и качестве главных локалей. Согласно актуальным рекомендациям Google, языковые варианты должны иметь отдельные URL, явные ссылки и взаимный `hreflang`; автоматические языковые редиректы мешают обнаружению. Страновая ориентация полезна только при реальном локальном различии, а массовые вариации ради поиска создают риск слабого или doorway-контента.

## Goals / Non-Goals

**Goals:**

- Создать поддерживаемую архитектуру для шести новых языков и приоритетных стран.
- Обеспечить отдельную поисковую идентичность, полезную локальную редактуру и технически согласованный языковой кластер.
- Сохранить продуктовую достоверность, доступность, производительность и GitHub Pages-совместимость.
- Выпускать локали независимо, но добавлять в публичный кластер только полностью проверенные версии.

**Non-Goals:**

- Не создавать в этом изменении тематические страницы для каждого языка.
- Не создавать варианты для всех стран, где употребляется английский, испанский, французский, немецкий или португальский.
- Не добавлять framework, template engine, сборщик, backend, аналитику, определение IP или автоматические редиректы.
- Не публиковать точные цены, отзывы, рейтинги или локальные юридические обещания без отдельного подтверждения.
- Не генерировать новые изображения только ради смены языка, если существующая графика не содержит текста.

## Decisions

### 1. Матрица первого релиза

| Рынок | Язык документа | URL | `hreflang` | `og:locale` |
|---|---|---|---|---|
| Россия / русскоязычная версия | `ru` | `/` | `ru` | `ru_RU` |
| Международная английская версия | `en` | `/en/` | `en`, `x-default` | `en_US` |
| Германия | `de` | `/de/` | `de-DE` | `de_DE` |
| Испания | `es` | `/es/` | `es-ES` | `es_ES` |
| Франция | `fr` | `/fr/` | `fr-FR` | `fr_FR` |
| Индия | `hi` | `/hi/` | `hi-IN` | `hi_IN` |
| Бразилия | `pt-BR` | `/pt-br/` | `pt-BR` | `pt_BR` |
| Турция | `tr` | `/tr/` | `tr-TR` | `tr_TR` |

Английский остаётся общим и `x-default`, как зафиксировано в правилах проекта. Для новых языков регион указывается в `hreflang`, потому что первая редактура ориентирована на конкретную страну. URL остаются короткими языковыми подкаталогами, кроме `/pt-br/`, где необходимо заранее различить бразильский и европейский варианты португальского.

Альтернатива — маршруты `/de-de/`, `/es-es/` и подобные. Она отклонена: сейчас для этих языков существует только один вариант, а регион уже однозначно выражен через `hreflang`. При появлении второго рынка решение пересматривается без переноса действующего URL.

### 2. Статические страницы с общей структурой, но самостоятельной редактурой

Каждая локаль реализуется как отдельный `index.html`, созданный на базе проверенной семантической структуры английского лендинга. CSS можно сначала оставить встроенным для минимального риска, но повторяемые стили должны оставаться синхронизированными. Текст, metadata, FAQ и JSON-LD редактируются как единый локальный документ, а не подставляются JavaScript.

Альтернатива — внедрить генератор страниц или framework. Она отклонена правилами проекта и создаёт несоразмерную инфраструктуру для шести статических страниц.

### 3. SEO-исследование является входным условием для текста

Для каждой страны перед написанием фиксируется краткий market brief:

- основной термин Tarot и варианты онлайн-расклада;
- естественное название AI-интерпретации и Telegram-бота;
- локальные названия ключевых раскладов;
- поисковое намерение главной страницы и 3–5 реальных вопросов FAQ;
- языковые табу, рискованные обещания и особенности оплаты;
- проверенные варианты `title`, `h1`, description и CTA.

Market brief хранится рядом с change-артефактами либо в рабочем документе до реализации. Финальный HTML содержит только полезную пользователю редактуру. Автоперевод допустим как черновик, но не как готовый к публикации текст.

Альтернатива — перевести английскую страницу и заменить ключевые слова. Она отклонена как недостаточная для локального намерения и качества.

### 4. Публикуемый hreflang-кластер обновляется атомарно

После готовности конкретной локали обновляются:

1. self-canonical и локальные metadata новой страницы;
2. полный взаимный набор `hreflang` на всех уже опубликованных основных страницах;
3. видимый переключатель языка;
4. соответствующие записи `sitemap.xml`;
5. JSON-LD `inLanguage`, URL и `@id`.

Незавершённые локали не рекламируются как доступные на сайте и не включаются в sitemap. Это позволяет выпускать по одной стране, не оставляя поломанных взаимных ссылок.

Альтернатива — сразу добавить ссылки на будущие URL. Она отклонена из-за 404, плохого UX и несогласованного кластера.

### 5. Языковой переключатель использует названия языков, а не только флаги

На мобильных и десктопных страницах используется компактный доступный список реальных ссылок с самоназваниями языков: Русский, English, Deutsch, Español, Français, हिन्दी, Português (Brasil), Türkçe. Флаг может быть декоративным дополнением, но не единственным обозначением, поскольку язык не равен стране.

Альтернатива — восемь отдельных кнопок в шапке. Она отклонена из-за перегрузки узких экранов и неоднозначности флагов.

### 6. Изображения переиспользуются

Главные изображения из `images/en/` не содержат локализованного текста и могут обслуживать новые страницы с переведёнными `alt`. Новые изображения создаются только при отдельной контентной потребности; исходные `images/referens.png` и `images/referens2.png` остаются неизменными.

### 7. Проверка выполняется на уровне страницы и всего графа

Локальные проверки включают HTML/JSON-LD parsing, точные CTA, существование assets, единственный `h1`, heading hierarchy, canonical/locale, отсутствие горизонтального overflow на обязательных ширинах и работу без JavaScript. Графовая проверка сравнивает все главные страницы и sitemap: каждая опубликованная альтернатива должна присутствовать взаимно и вести на реальный URL.

Языковая проверка отделена от технической: корректный HTML не подтверждает естественность текста.

## Risks / Trade-offs

- [Низкокачественный перевод снижает доверие и поисковую ценность] → Использовать market brief и обязательную финальную проверку носителем или квалифицированным редактором.
- [Массовое создание страновых дублей выглядит как doorway/масштабированный контент] → На первом этапе одна полезная локаль на новый язык и документированное обоснование для каждого будущего регионального варианта.
- [Восемь альтернатив легко рассинхронизировать] → Обновлять кластер атомарно и добавить автоматическую проверку canonical/`hreflang`/sitemap.
- [Встроенный CSS дублируется] → Не проводить архитектурный рефакторинг одновременно с локализацией; синхронизацию стилей проверять сравнением, а вынос CSS рассматривать отдельным изменением.
- [Хинди может получить смешанную или неестественную лексику] → Разрешить общеупотребительные заимствования только после локального исследования и проверить отображение деванагари выбранными шрифтами.
- [Один региональный вариант не покрывает всех носителей языка] → Сохранить возможность будущих вариантов, но не создавать их без данных и уникальной редакционной ценности.
- [Изменение всех основных страниц увеличивает риск регрессии] → Выпускать локали поэтапно и иметь откат через удаление новой страницы из кластера и восстановление предыдущих HTML/sitemap из Git.

## Migration Plan

1. Зафиксировать базовые проверки существующих `/` и `/en/`.
2. Для каждой страны подготовить market brief и локальную редактуру.
3. Реализовывать локали по одной, начиная с одного пилотного рынка.
4. После технической и языковой проверки добавлять готовую локаль атомарно во все `hreflang`, переключатели и sitemap.
5. Проверить весь опубликованный кластер через локальный HTTP-сервер на обязательных viewport.
6. Публиковать обычным механизмом GitHub Pages без изменения `CNAME`.
7. После публикации проверить реальные URL, canonical и sitemap; затем отправить sitemap и проинспектировать новые URL в Google Search Console и Bing Webmaster Tools.

Откат одной локали: убрать её ссылки из остальных страниц и sitemap, затем удалить или вернуть предыдущую версию её каталога через Git. Остальные локали продолжают работать.
